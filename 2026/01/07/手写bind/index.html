Function.prototype.bind简单实现：正确处理 new 操作符

Function.prototype.bind 是 JavaScript 中用于创建绑定特定 this 上下文的函数的重要方法。然而，很多简单的实现都会忽略一个关键场景：当返回的函数被 new 操作符调用时会发生什么？ 本文将深入分析这个问题，并提供完整的实现方案。

为什么出错？new 操作符的执行机制

new 操作符的执行顺序：

  1. 创建一个新对象
  2. 将新对象的 __proto__ 指向构造函数的 prototype
  3. 将构造函数的 this 绑定到新对象并执行
  4. 如果构造函数返回对象则返回该对象，否则返回新对象

问题根源分析

当执行 new a() 时：

  1. new 创建对象 b，并将 b.__proto__ 指向 a.prototype
  2. new 调用 a.call(b)，期望 a 内部的 this 指向 b
  3. 但 a 是 myBind 返回的函数，内部执行 me.apply(context, finalArgs)
  4. 由于 context 是 null，test 的 this 被强制指向 null
  5. 结果：b.name 是 undefined

核心矛盾：new 想让 this 指向新实例，但 bind 的实现强制使用 context。

解决方案：检测是否被 new 调用

需要区分两种情况：

  • 普通调用：使用 context（或 this）
  • new 调用：忽略 context，让 this 指向新实例

工作原理详解

普通调用 a()

  1. this 不是 F 的实例（this instanceof F 为 false）
  2. 执行 me.apply(context, finalArgs)
  3. test 的 this 指向 context（或全局对象）

new 调用 new a()

  1. new 创建对象，this 指向该对象
  2. 由于 bound.prototype = new F()，this.__proto__ 指向 F.prototype
  3. 因此 this instanceof F 为 true
  4. 执行 me.apply(this, finalArgs)（忽略 context）
  5. test 的 this 指向新实例

关键点总结

  • 数据结构：用中间构造函数 F 建立原型链，支持 instanceof 检测
  • 特殊情况消除：通过 instanceof 统一处理普通调用和 new 调用
  • 向后兼容：不影响普通 bind 的使用

总结

实现 bind 时，需要处理 new 调用的情况。通过中间构造函数建立原型链，用 instanceof 检测调用方式，确保 new 调用时 this 指向新实例，普通调用时使用绑定的 context。这样既满足规范，又保持向后兼容。
